# Onboard

The one-time setup for the AI OS. Run it once, after the folder and the project, gem, or custom GPT are connected. It goes in four steps: platform, paths, interview, populate, then a map. Ask one question at a time throughout. Reflect each answer back in a line before the next question.

## 1. Platform

Ask first: **which AI are you running this in, Claude, Gemini, or ChatGPT?** Read `_shared/platforms.md`. The answer changes how the OS is indexed, so settle it before anything else.

- **Claude.** Saves go to each path's `decisions/` as new dated files (the connector is create-only). The rollup `overview.md` is kept current by the user or regenerated by the audit. Skills are uploaded.
- **Gemini.** A Gem references about 10 files, so point it at the rollups only: `me.md`, `_shared/`, and one `overview.md` per active path. Keep `decisions/` for history but out of the Gem. Keep rollups tight.
- **ChatGPT.** Link the whole OS folder to a Project. Capture is manual into `inbox/`. Build the capability layer as a Custom GPT.

Tell the user, in one line, how their choice shapes the setup. Then move on.

## 2. Paths

Ask: **which areas do you want the OS to hold, business, personal, work, or more than one?** Only scaffold the ones they pick. For each chosen path you will write an `overview.md` and seed `references/` and `decisions/`. Do not create empty paths they did not ask for.

## 3. Interview

Ask these one at a time, scoped to the paths they chose. For each path, the spine is the same: who, what, how the work runs, what good looks like.

Shared, asked once:

- Who are you, and how do you want me to work with you? Peer or assistant, ask or guess, how much before checking in.
- Paste two or three real messages you have written, so I can learn your voice.
- What do you never want me to say or do.

Per path (ask only for chosen paths):

- What is this area, and what are you trying to do in it?
- Who else is involved, and who does what?
- Walk me through the main process here, start to finish.
- What do you keep doing by hand that you wish ran itself?
- What does a great week look like in this area?

## 4. Populate

From the answers, draft and save (new files, never overwriting an existing one). Show each one, get a yes, then save.

- `me.md` — who they are, how they want you to work, their voice, their anti-patterns. Shared across all paths.
- `<path>/overview.md` — the living rollup for each chosen path: who, what, the process, what good looks like. Keep it tight. This is the file Gemini leans on, so do not let it sprawl.
- A first note in `<path>/decisions/` if anything they said is a standing call worth recording.

State the boundary out loud and put it in `me.md`: what never goes in the OS. Sensitive personal data, client secrets, anything that should not sit in a folder an AI can read.

## 5. Map

Pull a first map of what the AI can actually see, so the OS starts grounded in reality and not just in answers. Read-only.

- Look across the connected tools: recent folders in Drive, the calendar shape, the CRM or boards if connected, mail.
- Draft `_shared/connections.md` recording what each tool holds and what you found in it. Tag each tool with the path it serves.
- Show the map. Flag anything missing: a tool they mentioned that is not connected, or a blind spot.

## Output

A short recap: the platform and how the OS is now indexed for it, what you saved and where, and the map of what you can see. Then point them at the weekly rhythm: "run my audit" and "level up".

## Rules

- One question at a time. Never dump the whole interview at once.
- Settle the platform first. It changes the file shape.
- Confirm each file before saving. The user owns their me.md and their overviews.
- Pull the map read-only. Do not change anything in the connected tools during setup.
- If a tool or path is not ready, note it and move on. Do not stall.
